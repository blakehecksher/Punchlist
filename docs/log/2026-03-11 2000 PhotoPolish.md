# 2026-03-11 2000 PhotoPolish

## TL;DR
- What changed: Rewrote photo rendering from `<img>` with `object-fit:cover` to `background-image` with `background-size`/`background-position`. Fixed crash on first drag. Added window-level drag listeners.
- Why: `object-fit:cover` only pans along the cropped axis — landscape photos in portrait cells couldn't pan left/right. Drag stopped when cursor left the photo cell.
- What didn't work: Several intermediate approaches hit React 19's strict lint rules (no setState in effects, no ref access during render, no self-referencing callbacks). Each was resolved.

## Full notes

### Problem 1: Single-axis panning
`object-fit: cover` + `object-position` only creates panning room along the axis where the image is cropped. A landscape photo in a portrait cell is cropped horizontally, so `object-position` only affected the X axis. At zoom levels >1, the oversized `<img>` element with `object-fit: cover` didn't create additional panning room because cover fills the element, not the container.

**Fix**: Replaced `<img>` element entirely with CSS `background-image` on the cell div:
- `background-size: {scale * 100}%` — at scale=1 the image width fills the cell. At scale=2 the image is twice the cell width.
- `background-position: {x}% {y}%` — pans freely in both directions at any zoom level.
- No `object-fit` involved — background properties handle sizing and positioning natively.

### Problem 2: Crash on first drag
`setLocalPos(prev => ...)` received `null` as `prev` because `localPos` was initialized to `null` (meaning "use prop value"). Spreading `null` and accessing `.scale` crashed the app.

**Fix**: Switched from nullable `localPos` to always-initialized `pos` state. Uses the `setState-during-render-with-guard` pattern (`if (photo !== photoKey)`) to reset when a new photo loads — no effects, no refs during render.

### Problem 3: Drag stopped at cell boundary
`onMouseMove` and `onMouseUp` were React event handlers on the cell div. When the cursor left the div, `onMouseLeave` fired `handleMouseUp` which killed the drag.

**Fix**: On `mousedown`, attach `mousemove` and `mouseup` to `window`. The `mouseup` handler removes both listeners. A `cleanupDrag` ref prevents listener leaks if `mousedown` fires again before `mouseup`.

### React 19 lint hurdles
1. `useEffect` + `setState` → "cannot call setState in effect" — fixed by removing the effect entirely
2. `useRef` read/write during render → "cannot access refs during render" — fixed by using state-based `photoKey` guard instead
3. Self-referencing `useCallback` → "cannot access variable before declaration" — fixed by inlining listeners inside `handleMouseDown`
