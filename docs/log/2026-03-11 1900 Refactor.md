## Full notes

### Photo crop/pan fix
The old approach used `object-fit: cover` on an `<img>` with `transform: translate() scale()`. This doesn't work because:
1. `object-fit: cover` crops the image to fill the box first
2. CSS `transform` then moves/scales the already-cropped result
3. For landscape photos in portrait cells, you lose top/bottom with no way to recover

New approach:
- Image sized to `width: {scale*100}%` / `height: {scale*100}%` with `object-fit: cover`
- `object-position: {x}% {y}%` controls the focal point (0-100% range)
- At scale=1, it's normal cover behavior. At scale>1, the image is oversized so panning reveals more area
- Dragging converts pixel deltas to percentage shifts, clamped to 0-100%
- Minimum scale is 1 (can't zoom out past cover)

### Photo position persistence
- IDB entries changed from plain `dataUrl` strings to `{ dataUrl, position }` objects
- Position = `{ scale, x, y }` where x/y are object-position percentages
- Backward-compatible: loading handles both legacy string format and new object format
- Position debounced (300ms) before persisting to avoid hammering IDB during drag

### useReducer
Replaced 10+ `setData(d => ({ ...d, ... }))` patterns with a single reducer. Actions: `load`, `setField`, `setSiteCondition`, `removeSiteCondition`, `addSiteCondition`, `updateItem`, `setPhoto`, `setPhotoPosition`, `addGeneralNote`, `removeGeneralNote`, `addRoomItem`, `removeRoomItem`, `setRoomName`, `addRoom`, `removeRoom`.

Helper `mapItem(data, id, fn)` finds an item by ID across both generalNotes and rooms, avoiding repeated map-within-map patterns.
